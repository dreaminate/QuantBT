## 提示词正文（贴这整段）

你之前给 QuantBT 出过完整 A-J 方案，又对 §D / §G / §H / §J 做了深化补丁（约 9000 字，质量合格）。这一轮请用**完全相同的深化标准**对剩下的 §A / §B / §C / §E / §F / §I 六节打补丁，不重排原 A-J 结构，只用于替换这六节。

### 通用要求（与 §D/§G/§H/§J 深化版一致）

- 每节深化到至少 800 字；整版总字数 ≥ 5000 字。
- 每节带 (a) 具体行动条目 (b) 度量指标 (c) 风险 (d) 备选方案。
- 数字必须有推导依据，不许凭感觉（"+20% 留存" 这种禁说）。
- 推荐必须带对标产品名 + 它的具体机制（不要写"做得更好"）。
- 不许 emoji、不许营销话术、不许 Bloomberg-style 类套话。
- 不许推荐 Web3 / 区块链 / 元宇宙 / NFT。
- 风险章节不许"市场风险、运营风险、技术风险"三件套。

### 章节具体深化要求

#### §A. 商业模式 & 产品定位（深化）

- 用一张表对比 5-7 家对标产品（聚宽、BigQuant、QuantConnect、米筐、vnpy、Numerai、Quantopian 已停服可参考），每家 **必须含**：
  - 上线时间 / 现状（活跃 / 萎缩 / 已停服）
  - 主力变现方式（订阅 / 数据出售 / 比赛 / 抽佣 / 培训）
  - 核心用户规模区间
  - **他们的一个具体决策错误**（QuantBT 要避开）
- **定价采用人民币（¥）**，目标用户是中文圈。给出三档及具体数字（¥），每档绑：(1) 包含功能列表 (2) 类比对标产品定价 (3) 锚点（"是 BigQuant Pro 的 X%"）
- 商业模式打分 5 种以上：开源 / 个人订阅 / 团队订阅 / 带单 GMV 抽佣 / 数据集出售 / 培训课程 / 比赛悬赏。每种给 (a) 可行性 1-10 (b) 启动门槛人时 (c) 法律风险 1-10
- "独特价值主张"必须 5 秒能被聚宽用户理解（用一句不超过 30 字的话）+ 给 3 个候选 + 推荐其中一个
- 给出从注册到付费的"信任阶梯"（trust ladder）至少 5 级，每级给一个具体功能解锁

#### §B. 目标客户细分 + 关键场景（深化）

- P0 / P1 / P2 每类用户给 **真名虚构 persona**（"32 岁北京后端张磊，3 年 Python，做过两个加密策略亏了 8w"），每个 persona 含：
  - 收入区间
  - 已用工具栈
  - 当前最大痛点（具体到一句话）
  - 第一次进 QuantBT 期望解决什么
  - 一周内放弃的 trigger
  - 半年后变成什么样的用户（晋级/流失/付费）
- "一周典型使用流"必须**绑定真实时点**（"周一晚 22:00 跑回测因为白天上班"），不许只列页面
- 五阶段 funnel 每段给：(1) 预期转化率 (2) 主要 drop-off 原因 3 条 (3) 干预的具体功能名（不许写"提示用户"这种话）
- 写出 P0 用户的 "Aha 时刻"——他第一次在 QuantBT 体验到聚宽给不了的什么瞬间？至少 3 个候选 Aha
- 把 P0/P1/P2 三类用户的 LTV / CAC 估算给出（中文圈量化用户的合理假设）

#### §C. 12 周产品路线图（深化）

- 12 周每周一行不够；每周必须给：
  - **主题名**（4-8 字）
  - **能提交的 git tag**（如 v0.8.4 / v0.8.4.1）
  - **3-5 条 user-facing 改动**
  - **2-3 条后端基建**
  - **测试基线 delta**（从当前 261 起算，到 12 周后多少）
  - **3 个完成判定命令**（pytest / tsc / curl 之类）
  - **RICE 拆 4 维**（Reach 1-10 / Impact 1-10 / Confidence 1-10 / Effort 人时）
- 周与周之间必须显式标 dependency（"Week 5 需要 Week 3 的 conversation 表先落地"）
- 第 12 周必须能 ship v0.9.0；给出 v0.9.0 的定义（功能边界 + 不做什么）
- 必须有至少 1 周是"还债 / 重构 / 不出新功能"周——告诉我哪一周，做什么

#### §E. 用户粘性最高 ROI 的 5 个杠杆点（深化）

- 5 个杠杆点每个给：
  - 名字（不超过 8 字）
  - 解决的具体粘性问题（基于 §B funnel 的哪个 drop-off）
  - 实现复杂度 1-5 + 对应的 v0.8.x 版本号
  - **预期 7 日留存提升幅度的推导**（"基于聚宽公开数据 / Cohort 假设 X，预计提升 Y%"）；不许凭空给百分比
  - 失败信号（如果上线后看到什么数字就该回滚）
  - 至少一个对标产品该功能的实现细节（Numerai Bounty / Discord Stage / Linear 进度等）
- 至少 2 个杠杆点是**社交向**，至少 1 个是**Gamification / 学习路径**，至少 1 个是**强制阻断式安全**（"没做 X 不让你 Y"是粘性手段，因为用户必须回来做 X）

#### §F. 对标竞品的差异化漏洞（深化）

- 写一份 "用户为什么从聚宽 / BigQuant / QuantConnect 离开来 QuantBT"，每个来源给 **3 条具体理由**（不许写"因为更好用"），每条理由对应一个 QuantBT 已有 / 待做的功能
- 反过来写 "用户为什么可能从 QuantBT 离开"，至少 5 条，每条给 1 个对策
- 给出 QuantBT vs 各家的 **feature matrix 表格**（数据接入 / 实盘 / 社区 / 教学 / 透明度 / 价格 / 安全 / DIY 8 维），每维 1-5 分，QuantBT 列单独算总分对比
- 推荐 3 个 QuantBT 必须比聚宽强 5 分以上才能立足的维度（说人话：你的护城河是什么）

#### §I. 团队 / 资源 minimum viable（深化）

- "单人 + AI 能撑到哪一步" 给一张分阶段表：
  - 用户数 0-50: 配置（人时/周 / Claude $ / 基础设施 $）
  - 50-500
  - 500-5000
  - 5000+
  - 每档列：必须自己干的 / 可外包的 / 必须招的角色 + 招的 trigger
- 第一个全职合伙人画像：必须的技能 3 条 + 加分项 3 条 + 不要的人特征 3 条 + 期权/薪资合理区间
- AI 协作工作流改进建议至少 5 条，每条给 (a) 当前耗时 (b) 优化后耗时 (c) 实施成本
- 内容/社区运营：列出 5 种内容形式（教程 / 案例 / 直播 / 文档 / Twitter），每种给"用户写 / Claude 写 / 第三方写"的可行性 + ROI
- Burnout 风险：单人项目 6-12 个月最大的崩盘原因 3 条 + 预防措施

### 输出格式

每节用 `## §X. ...（深化版）` 标题。仍然用 markdown 表格能用就用，不许大段散文。各节结尾给一行 "TL;DR for this section"，3-5 条 bullet。

整版结尾给一份 **全文 TL;DR 10 条**——这是 GPT Pro 该交的最后总结，10 条要互相不冗余、每条独立可被产品决策引用。
