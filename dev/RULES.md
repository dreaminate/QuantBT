# RULES · OS 通用铁律【开发os级别】（勿改 · 跨项目一致）

> 本文件是这套开发 OS 的「宪法」,任何用本 OS 的项目都继承它。**项目自己的红线写在 `RULES.project.md`**。
> 删掉本文件任一条 = 你用的就不再是这套 OS。

<!-- 【开发os级别】勿改 · clone 自 dev-os。本项目红线写 RULES.project.md,不写这里。 -->

## 0. 哲学
处处最专业最严谨;用户有困难派 agent 辅助,而不是降低标准。

## 1. 复用 + 单一源
- 一个能力一个源,消灭重复实现;查询走索引、防篡改走 append-only。
- 复用现有模块/范式,不重造轮子。

## 2. 对抗测试标准（绝不妥协）
不是覆盖率,是「**种一个已知的坏,门必须抓住,否则门是纸做的**」。每部件配:针对其失效模式的探针 + 变形测试 + 幂等 + 措辞/裁决核对。验证还要**变异测试**(种已知 bug,门全杀)。

## 3. 诚实纪律
- 🟡「声称但未验证」≠ ✅「已建并验证」——状态文件绝不假绿灯。
- 裁决说「证据充分/不足 + 适用域 + 没验证的」,**不说「可信/安全/保证」**。

## 4. 工程红线
- **不破坏现有测试基线**(具体数目以 STATE 为准,**绝不把易变数字写进慢变文件** = 防漂)。
- 改现有文件「**扩展不替换**」。
- **不擅自 commit/push**——用户明说才提交。
- 决策记录 `DECISIONS.md` **append-only**,锁定不改既往。

## 5. 致命错误即停工
项目把「哪些是致命错误」定义在 `RULES.project.md`(典型:不可逆操作 / 动钱 / 安全不变量被削弱 / 数据泄露)。开发中**引入或削弱**任一项 → **立即停工、报告、不擅自继续**。

## 6. 审计 / 复核纪律（防过度审计 + 防漏审）
先立框架,再挑刺:
1. **大结构**:先定分类——有哪些类别、每处应当装什么(`RULES.project` 装项目红线、`GOAL` 装终态…)。
2. **小结构**:再判归属——此处是否装着它该装的那类(**项目内容出现在项目文件里 = 对,不是漂移**)。
3. **细节**:最后才行级核对——在「此处该是什么」的框架内,挑真正偏离的。

铁律:**先确立「此处应是什么」,再判「是否偏离」;没有框架,不给「漂移/错误」定性。**
反向防呆:框架是**假设、不是真理**。若某细节真的违反 OS 不变量(不只是不合预期),**用它推翻/修正框架,别把真问题合理化**——防过度审计 ≠ 放过真问题。
